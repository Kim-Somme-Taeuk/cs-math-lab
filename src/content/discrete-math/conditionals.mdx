## 왜 배우나

조건문은 컴퓨터공학에서 매우 자주 등장합니다. 함수의 입력 조건, 테스트의 기대 결과,
알고리즘 증명의 "만약 ...라면 ..." 구조가 모두 조건문입니다.

수학의 조건문 $P \to Q$를 이해하면 코드에서 어떤 조건이 깨졌는지 더 정확하게 설명할 수
있습니다. 특히 "전제가 성립할 때 무엇이 보장되는가"를 분리해서 읽는 능력이 생깁니다.

```ts
// 사용자가 관리자라면 관리자 페이지에 접근할 수 있다.
if (user.role === "admin") {
  showAdminPage();
}
```

이 문장은 "관리자가 아니면 절대 접근할 수 없다"까지 자동으로 말하지는 않습니다. 조건문은
앞조건이 참인 경우에 뒤결론이 따라오는지를 검사합니다.

## 먼저 감 잡기

$P \to Q$는 "P라면 Q이다"라는 약속입니다.

예를 들어 P가 "비가 온다", Q가 "우산을 가져간다"라면 조건문은 "비가 오면 우산을
가져간다"입니다. 이 약속이 실제로 깨지는 경우는 비가 왔는데 우산을 가져가지 않은
경우뿐입니다.

다른 예도 같습니다.

- "회원이면 할인된다"는 회원인데 할인되지 않을 때 깨집니다.
- "배열이 정렬되어 있으면 이진 탐색을 적용할 수 있다"는 정렬된 배열인데 이진 탐색 조건을
  만족하지 못할 때 깨집니다.
- "입력이 빈 문자열이면 에러 메시지를 보여 준다"는 빈 문자열인데 에러 메시지가 없을 때
  깨집니다.

## 핵심 정의

조건문 $P \to Q$는 P가 참이고 Q가 거짓일 때만 거짓입니다. 나머지 경우는 모두 참입니다.

| P | Q | $P \to Q$ |
| --- | --- | --- |
| 참 | 참 | 참 |
| 참 | 거짓 | 거짓 |
| 거짓 | 참 | 참 |
| 거짓 | 거짓 | 참 |

논리식으로는 다음과 같이 바꿔 쓸 수 있습니다.

$$
P \to Q \equiv \lnot P \lor Q
$$

즉 "P이면 Q"는 "P가 아니거나 Q이다"와 같은 진리값을 갖습니다. 코드에서는 다음 두 조건이
같은 의미로 쓰일 수 있습니다.

```ts
const valid = !isMember || hasDiscount;
```

## 자주 헷갈리는 지점

P가 거짓인데 조건문이 참인 경우가 가장 낯설게 느껴집니다. 하지만 조건문은 "P인 상황에서
Q가 따라오는가"를 검사합니다. P가 일어나지 않았다면 아직 약속 위반 사례가 없습니다.

예를 들어 "회원이면 할인된다"라는 규칙을 생각해 봅시다.

- 회원이고 할인됨: 규칙을 지켰습니다.
- 회원인데 할인 안 됨: 규칙을 어겼습니다.
- 회원이 아닌데 할인됨: 이 규칙만으로는 위반이라고 말할 수 없습니다.
- 회원이 아니고 할인 안 됨: 이 규칙만으로는 위반이 아닙니다.

조건문을 "P일 때만 Q이다"와 혼동하면 안 됩니다. "P이면 Q"는 P가 Q의 충분조건이라는
뜻이지, Q가 되려면 반드시 P여야 한다는 뜻은 아닙니다.

프로그래밍의 사전조건도 비슷합니다. 특정 입력 범위에서만 결과를 보장하는 함수는 그 입력
범위 밖에 대해서는 다른 규칙을 적용할 수 있습니다. 그래서 조건문을 읽을 때는 "이 문장이
보장하는 범위"와 "아직 말하지 않은 범위"를 구분해야 합니다.

### 예제를 해석하는 순서

조건문을 볼 때는 다음 질문을 차례대로 던집니다.

1. 앞조건 P는 무엇인가?
2. 뒤결론 Q는 무엇인가?
3. P가 참인데 Q가 거짓인 반례가 있는가?
4. Q가 참인 이유가 반드시 P 때문이라고 착각하고 있지는 않은가?

예를 들어 "관리자라면 삭제할 수 있다"는 다음처럼 읽습니다.

```txt
P: 관리자이다.
Q: 삭제할 수 있다.
깨지는 경우: 관리자이지만 삭제할 수 없다.
아직 말하지 않은 것: 관리자가 아닌 사람이 삭제할 수 있는지 여부
```

이 마지막 줄이 중요합니다. 조건문은 보장하는 범위와 보장하지 않는 범위를 함께 읽어야
오해가 줄어듭니다.



## 코드로 읽기

```ts
function binarySearch(sortedNumbers: number[], target: number) {
  // 전제 P: sortedNumbers는 오름차순으로 정렬되어 있다.
  // 결론 Q: 반환값은 target의 위치이거나, 없다는 사실을 올바르게 나타낸다.
}
```

알고리즘 설명에서 전제가 P이고 결론이 Q라면, 정확성 주장은 보통 $P \to Q$ 형태를 갖습니다.

입력 검증 코드도 같은 방식으로 읽을 수 있습니다.

```ts
function submitEmail(email: string) {
  if (email.length === 0) {
    return { ok: false, message: "이메일을 입력하세요." };
  }

  return { ok: true };
}
```

여기서 핵심 조건문은 "이메일이 비어 있다면 에러 메시지를 반환한다"입니다. P는
`email.length === 0`, Q는 `ok`가 `false`이고 메시지가 있다는 사실입니다. 테스트를 만들 때는
P가 참인 입력을 넣고 Q가 따라오는지 확인하면 됩니다.

## 전공에서 만나는 장면

조건문은 다음처럼 작은 사례로 자주 나타납니다.

- 알고리즘 정확성: "루프가 끝났다면 모든 원소를 검사했다" 같은 주장을 세웁니다.
- 타입 가드: "`typeof value === 'string'`이면 `value.toUpperCase()`를 호출할 수 있다"고
  추론합니다.
- API 계약: "토큰이 만료되었으면 401을 반환한다"처럼 서버 응답 조건을 명시합니다.
- 테스트 설계: 특정 입력 조건 P를 만들고 기대 결과 Q를 검증합니다.
- 보안 정책: "관리자이면 삭제할 수 있다"와 "삭제할 수 있으면 관리자이다"를 구분합니다.

"어떤 입력이면 어떤 결과가 보장된다"는 문장을 다루는 능력은 전공 과목 전반에 필요합니다.
