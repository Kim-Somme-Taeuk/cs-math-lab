## 왜 배우나

점근적 분석은 <strong>입력 크기가 커질 때 알고리즘 비용이 어떻게 증가하는지</strong> 보기 위해 배웁니다.
실행 시간을 초 단위로 맞히는 기술이 아니라, 코드의 반복 구조와 자료구조 선택이 큰 입력에서 어떤 차이를
만드는지 읽는 방법입니다.

컴퓨터공학에서는 이 감각이 바로 쓰입니다.

- 코딩테스트: 입력 제한과 시간 제한을 보고 가능한 풀이를 고릅니다.
- 서버 성능: 요청 수나 데이터 수가 늘어날 때 병목이 어디서 생기는지 예측합니다.
- 자료구조 선택: 배열, 해시셋, 힙, 그래프 표현 중 어떤 구조가 비용을 줄이는지 비교합니다.

이 장의 핵심은 Big-O 정의를 엄밀히 증명하는 것이 아닙니다. <strong>코드를 보고 성장률을 읽는 법</strong>을
익히는 것입니다.

## 먼저 감 잡기

분석을 시작하기 전에 먼저 n이 무엇인지 정해야 합니다. <strong>입력 크기 n은 비용을 결정하는 기준</strong>입니다.

문제마다 n은 다르게 정해질 수 있습니다.

- 리스트 문제: 리스트 길이
- 문자열 문제: 문자열 길이
- 그래프 문제: 정점 수 V와 간선 수 E
- 격자 문제: 행 수와 열 수, 또는 전체 칸 수

예를 들어 리스트에서 중복을 찾는 문제라면 보통 n은 리스트 길이입니다.

```python
def print_once(values):
    for value in values:
        print(value)

def print_pairs(values):
    for left in values:
        for right in values:
            print(left, right)
```

`print_once`는 원소가 n개면 대략 n번 일합니다. `print_pairs`는 각 원소마다 다시 n개를 훑으므로 대략
$n \times n$번 일합니다. n이 10이면 10과 100의 차이지만, n이 10000이면 10000과 100000000의 차이가 됩니다.

점근적 분석은 이런 차이를 $O(n)$과 $O(n^2)$처럼 입력 크기 기준으로 구분합니다.

## 핵심 정의

### 시간복잡도

<strong>시간복잡도는 실행 시간이 입력 크기에 따라 어떻게 증가하는지 표현하는 방법</strong>입니다. 여기서 실행
시간은 실제 초 단위 시간이 아니라, 비교, 대입, 반복, 함수 호출 같은 연산량의 성장률에 가깝습니다.

예를 들어 원소 n개를 한 번씩 더하면 $O(n)$이고, 모든 쌍을 비교하면 $O(n^2)$에 가깝습니다.

### 공간복잡도

<strong>공간복잡도는 추가 메모리가 입력 크기에 따라 어떻게 증가하는지 표현하는 방법</strong>입니다.

다음처럼 입력 외에 새로 만드는 공간을 봅니다.

- 새 배열이나 리스트
- 해시맵, 해시셋
- 큐, 스택
- 재귀 호출 스택

예를 들어 리스트를 한 번 훑으며 합만 저장하면 추가 공간은 $O(1)$입니다. 반대로 원소를 모두 새 리스트에
복사하면 추가 공간은 $O(n)$입니다.

### Big-O

Big-O는 비용 증가를 상한 중심으로 표현하는 표기입니다.

$$
T(n) = O(f(n))
$$

입문 단계에서는 이 식을 "입력이 커질 때 $T(n)$의 큰 증가 모양을 $f(n)$으로 잡는다" 정도로 읽으면 충분합니다.
정확한 실행 시간을 말하는 도구가 아니라, 성장률을 비교하는 도구입니다.

대표 성장률은 다음 순서로 자주 만납니다.

| 표기 | 느낌 | 흔한 예 |
| --- | --- | --- |
| $O(1)$ | 입력 크기와 거의 무관 | 첫 원소 접근 |
| $O(\log n)$ | 매번 절반씩 줄어듦 | 이진 탐색 |
| $O(n)$ | 한 번 훑음 | 배열 합, 선형 탐색 |
| $O(n \log n)$ | 나누고 각 단계에서 훑음 | 효율적인 정렬 |
| $O(n^2)$ | 모든 쌍을 봄 | 이중 반복 비교 |
| $O(2^n)$ | 선택지가 매 단계 두 배 | 부분집합 완전 탐색 |

### 상수와 낮은 차수 항 제거

Big-O를 읽을 때는 큰 입력에서 가장 빠르게 커지는 항을 봅니다.

$$
3n + 10 = O(n)
$$

$$
n^2 + n = O(n^2)
$$

상수 3이나 더해진 10이 실제 실행에서 아무 의미 없다는 뜻은 아닙니다. 다만 입력이 커질수록 전체 성장 모양은
가장 빠르게 커지는 항이 지배합니다.

## 자주 헷갈리는 지점

<strong>$O(2n)$처럼 상수를 남기지 않습니다.</strong> $2n$, $3n + 10$, $100n$은 모두 큰 모양으로는 $O(n)$입니다.

<strong>반복문 하나가 항상 $O(n)$인 것은 아닙니다.</strong> 반복문이 입력 크기와 무관하게 10번만 돌면 $O(1)$입니다.
반대로 반복문 안에서 매번 절반씩 줄어드는 구조라면 $O(\log n)$일 수 있습니다.

<strong>이중 반복문이 항상 $O(n^2)$인 것도 아닙니다.</strong> 안쪽 반복문이 매번 n번 도는지, 전체 합으로 몇 번
도는지 확인해야 합니다. 다만 모든 쌍을 비교한다면 보통 $O(n^2)$에 가깝습니다.

<strong>최선 시간만 보고 빠르다고 판단하면 위험합니다.</strong> 찾는 값이 첫 번째에 있으면 선형 탐색도 $O(1)$처럼
끝날 수 있습니다. 하지만 값이 없거나 마지막에 있으면 n개를 모두 봅니다.

시간을 말할 때는 최선, 평균, 최악을 구분합니다.

- 최선 시간: 가장 운 좋은 입력에서의 비용
- 평균 시간: 보통 입력을 가정했을 때의 비용
- 최악 시간: 가장 오래 걸리는 입력에서의 비용

입문 단계에서는 먼저 <strong>최악 시간</strong> 위주로 분석하면 안전합니다. 최악 시간은 시간 제한과 성능 보장을
생각할 때 특히 중요합니다.

## 직접 실험하기

실제로 n을 키워 보며 성장률을 비교합니다.

| n | $n$ | $n \log_2 n$ | $n^2$ |
| --- | ---: | ---: | ---: |
| 10 | 10 | 약 33 | 100 |
| 100 | 100 | 약 664 | 10000 |
| 1000 | 1000 | 약 9966 | 1000000 |

처음에는 $n \log n$과 $n^2$가 모두 커 보입니다. 하지만 n이 커질수록 $n^2$는 훨씬 빠르게 벌어집니다.
이 차이 때문에 입력이 큰 정렬 문제에서는 보통 $O(n^2)$ 정렬보다 $O(n \log n)$ 정렬을 먼저 고려합니다.

## 코드로 읽기

코드에서 시간복잡도를 읽을 때는 문법보다 입력 크기에 따라 반복 횟수가 어떻게 바뀌는지 봅니다.

### 첫 원소 접근: $O(1)$

```python
def first_value(values):
    return values[0]
```

리스트 길이가 커져도 첫 원소 접근은 한 번입니다. 그래서 시간복잡도는 $O(1)$입니다.

### 배열 합: $O(n)$

```python
def sum_values(values):
    total = 0

    for value in values:
        total += value

    return total
```

원소 n개를 한 번씩 봅니다. 시간복잡도는 $O(n)$이고, 추가로 저장하는 값은 `total` 하나라서 추가 공간은
$O(1)$입니다.

### 이진 탐색: $O(\log n)$

```python
def binary_search(sorted_values, target):
    left = 0
    right = len(sorted_values) - 1

    while left <= right:
        mid = (left + right) // 2

        if sorted_values[mid] == target:
            return True

        if sorted_values[mid] < target:
            left = mid + 1
        else:
            right = mid - 1

    return False
```

이진 탐색은 한 번 비교할 때마다 남은 범위를 절반으로 줄입니다. 그래서 $O(\log n)$입니다. 단, 이 분석은
입력이 정렬되어 있다는 전제가 있을 때만 맞습니다.

### 모든 쌍 비교: $O(n^2)$

```python
def has_duplicate_slow(values):
    for i in range(len(values)):
        for j in range(i + 1, len(values)):
            if values[i] == values[j]:
                return True

    return False
```

최악의 경우 거의 모든 서로 다른 쌍을 비교합니다. 쌍의 개수는 $n^2$에 비례하므로 최악 시간복잡도는
$O(n^2)$입니다.

해시셋을 쓰면 시간과 공간의 tradeoff가 생깁니다.

```python
def has_duplicate_fast(values):
    seen = set()

    for value in values:
        if value in seen:
            return True

        seen.add(value)

    return False
```

평균적으로 해시셋 조회와 추가를 상수 시간으로 보면 시간복잡도는 $O(n)$입니다. 대신 `seen`에 최대 n개를
저장하므로 추가 공간복잡도는 $O(n)$입니다.

## 전공에서 만나는 장면

점근적 분석은 다음 장면에서 기본 언어처럼 쓰입니다.

- 코딩테스트 시간 제한: 입력 제한을 보고 $O(n^2)$ 풀이가 가능한지 판단합니다.
- 정렬 알고리즘 비교: $O(n^2)$ 정렬과 $O(n \log n)$ 정렬의 차이를 설명합니다.
- 해시셋/배열 선택: 조회를 빠르게 하려고 추가 메모리를 쓰는 선택을 이해합니다.
- 재귀와 점화식: 재귀 호출이 얼마나 쌓이는지 식으로 표현합니다.
- 그래프 알고리즘 비용 분석: 정점 수 V와 간선 수 E를 기준으로 DFS/BFS 비용을 읽습니다.

입력 제한이 $n = 100000$인데 모든 쌍을 보는 $O(n^2)$ 풀이를 쓰면 대략 100억 번 수준의 비교가 필요할 수
있습니다. 반대로 $O(n \log n)$이나 $O(n)$ 풀이를 찾으면 같은 문제도 현실적인 시간이 됩니다.
