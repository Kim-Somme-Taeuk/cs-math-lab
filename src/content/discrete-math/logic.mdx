## 왜 배우나

컴퓨터는 많은 판단을 참과 거짓으로 처리합니다. `if`, `while`, 필터 조건, 권한 검사,
테스트 코드의 기대값은 모두 어떤 문장이 참인지 거짓인지 묻는 구조입니다.

명제 논리는 이런 판단을 수학적으로 다루는 가장 작은 단위입니다. 코드를 더 정확하게
읽고, 조건이 언제 깨지는지 설명하려면 먼저 "판단 가능한 문장"을 작은 단위로 나누는
습관이 필요합니다.

예를 들어 다음 요구사항은 한 문장처럼 보이지만 실제로는 여러 명제가 섞여 있습니다.

- "사용자가 로그인했고, 이메일 인증을 마쳤으며, 정지된 계정이 아니면 글을 쓸 수 있다."
- "상품이 품절이 아니거나 예약 판매 중이면 구매 버튼을 보여 준다."
- "점수가 60점 이상이고 지각이 3회 미만이면 수료 처리한다."

이 문장들을 작은 명제로 나누면 코드 조건식, 테스트 케이스, 예외 상황을 더 차분하게
다룰 수 있습니다.

## 먼저 감 잡기

명제는 참 또는 거짓을 정할 수 있는 문장입니다.

- "7은 홀수이다"는 참인 명제입니다.
- "2 + 2 = 5"는 거짓인 명제입니다.
- "현재 사용자는 로그인했다"는 실행 시점의 상태를 보면 참/거짓을 정할 수 있으므로 명제입니다.
- "이 코드는 좋은가?"는 기준이 없으면 참/거짓을 정하기 어려워 명제로 보기 어렵습니다.
- "버튼을 예쁘게 만들어라"는 명령문이므로 명제가 아닙니다.

프로그래밍에서는 실행 시점의 상태가 정해져 있을 때 참/거짓으로 평가되는 조건식을 명제처럼
다룰 수 있습니다. 먼저 상황을 말로 나누어 봅시다.

- P: 나이가 19세 이상이다.
- Q: 역할이 관리자이다.
- R: 역할이 편집자이다.
- 전체 판단: P이고, Q 또는 R이다.

Python으로 쓰면 다음처럼 읽을 수 있습니다.

```python
is_adult = age >= 19
is_admin = role == "admin"
is_editor = role == "editor"

has_permission = is_admin or is_editor
can_access = is_adult and has_permission
```

여기서 `is_adult`, `is_admin`, `is_editor`는 현재 변수 값이 주어지면 참 또는 거짓으로
평가되는 작은 판단입니다. `can_access`는 이 판단들을 `and`, `or`로 조합한 복합 명제처럼
읽을 수 있습니다. TypeScript나 JavaScript에서는 같은 구조를 `&&`, `||`로 표현하지만, 이
장에서는 수학적 연결을 보기 위해 Python을 먼저 사용합니다.

## 핵심 정의

명제는 참 또는 거짓 중 하나의 진리값을 갖는 문장입니다. 두 명제 P, Q가 있을 때 기본
논리 연산은 다음과 같습니다.

| 연산 | 의미 | 참이 되는 경우 |
| --- | --- | --- |
| $P \land Q$ | AND | P와 Q가 모두 참 |
| $P \lor Q$ | OR | P 또는 Q 중 하나 이상 참 |
| $\lnot P$ | NOT | P가 거짓 |

복합 명제는 작은 명제를 결합해서 만든 명제입니다.

- $P \land Q$: "P이고 Q이다"
- $P \lor Q$: "P이거나 Q이다"
- $\lnot P$: "P가 아니다"
- $(P \land Q) \lor R$: "P와 Q가 모두 참이거나, R이 참이다"

논리식은 한 번에 읽기보다 가장 안쪽 판단부터 읽는 편이 안전합니다. 예를 들어
$(P \land Q) \lor R$은 "P와 Q가 모두 참"이라는 한 덩어리와 "R이 참"이라는 다른 가능성을
OR로 연결한 식입니다. 반면 $P \land (Q \lor R)$은 P가 반드시 참이어야 하고, Q나 R 중
하나가 추가로 참이어야 합니다.

이 차이는 코드에서도 바로 드러납니다. 예를 들어 입장 규칙을 다음 두 방식으로 나눌 수
있다고 해 봅시다.

- A: 회원이고 입장권이 있거나, 게스트이면 입장할 수 있다.
- B: 회원이고, 입장권이 있거나 게스트이면 입장할 수 있다.

```python
can_enter_a = (is_member and has_pass) or is_guest
can_enter_b = is_member and (has_pass or is_guest)
```

첫 번째 조건은 게스트만 참이어도 입장할 수 있습니다. 두 번째 조건은 게스트 여부와 상관없이
회원이어야 합니다. 괄호는 단순한 표기가 아니라 정책을 바꾸는 장치입니다.

## 자주 헷갈리는 지점

일상어의 "또는"은 가끔 둘 중 하나만 고르는 느낌이지만, 수학의 OR과 많은 프로그래밍
언어의 논리 OR은 보통 둘 다 참인 경우도 참으로 봅니다. "쿠폰이 있거나 포인트가 있으면 할인 가능"이라는
규칙에서 쿠폰과 포인트가 둘 다 있어도 할인 가능인 것과 같습니다.

명제와 의견도 자주 헷갈립니다.

- "이 함수는 10ms 안에 끝난다"는 측정 기준이 있으므로 명제입니다.
- "이 함수는 빠르다"는 빠름의 기준이 없으면 명제로 쓰기 어렵습니다.
- "사용자 이름은 비어 있지 않다"는 명제입니다.
- "사용자 이름을 잘 입력해라"는 명령이므로 명제가 아닙니다.

또 하나의 흔한 실수는 긴 조건식을 한 번에 읽으려는 것입니다. "로그인했고, 이메일 인증을
마쳤으며, 정지된 계정이 아니면 글을 쓸 수 있다"는 문장은 다음처럼 나뉩니다.

- P: 로그인했다.
- Q: 이메일 인증을 마쳤다.
- R: 정지된 계정이다.
- 전체 판단: P이고 Q이고 R이 아니다.

```python
can_publish = (
    is_logged_in
    and has_verified_email
    and not is_suspended
)
```

초보자는 이 코드를 "글을 쓸 수 있는가?"라고만 읽기 쉽습니다. 하지만 디버깅할 때는
`is_logged_in`, `has_verified_email`, `is_suspended`를 각각 따로 확인해야 어느 조건 때문에
실패했는지 알 수 있습니다.

## 코드로 읽기

논리 연산은 조건문을 구성하는 재료입니다. 먼저 상황을 자연어로 씁니다.

- P: 사용자가 로그인했다.
- Q: 사용자가 관리자 권한을 가졌다.
- 전체 판단: P이고 Q이면 관리자 화면을 보여 준다.

```python
if is_logged_in and has_admin_role:
    show_admin_panel()
```

여기서 `is_logged_in`과 `has_admin_role`은 각각 현재 상태에 대한 참/거짓 판단이고, 전체 조건은
두 판단의 AND입니다.

조금 더 복잡한 요구사항도 같은 방식으로 분해할 수 있습니다.

```python
is_admin = role == "admin"
is_editor = role == "editor"

has_write_permission = is_admin or is_editor

can_write_post = (
    is_logged_in
    and has_write_permission
    and not is_suspended
)
```

- P: 사용자가 로그인했다.
- Q: 사용자는 관리자이거나 편집자이다.
- R: 사용자는 정지된 계정이다.
- 전체 조건: $P \land Q \land \lnot R$

Python 코드의 `and`는 $\land$, `or`는 $\lor$, `not`은 $\lnot$에 대응합니다. 이렇게 바꾸면
테스트 케이스도 자연스럽게 나옵니다. P, Q, R 중 하나씩 값을 바꿔 보면서
어느 조합에서 글쓰기가 허용되어야 하는지 확인할 수 있습니다.

## 전공에서 만나는 장면

명제 논리는 전공 과목의 큰 제목보다 더 작은 장면에서 반복됩니다.

- 자료구조: 스택이 비어 있는가, 큐가 가득 찼는가, 방문한 정점인가를 판단합니다.
- 데이터베이스: `WHERE active = true AND role = 'admin'`처럼 행을 남길 조건을 만듭니다.
- 테스트: "입력이 빈 문자열이면 에러가 난다"처럼 기대 결과를 참/거짓으로 검증합니다.
- 보안: 로그인 여부, 권한, 토큰 만료 여부를 조합해 접근 가능성을 판단합니다.
- 알고리즘: 반복문이 계속될 조건과 종료될 조건을 명제로 표현합니다.

복잡한 시스템도 결국 작은 판단들의 조합으로 동작합니다. 명제를 잘 나누면 코드의 의도를
설명하기 쉬워지고, 실패한 조건을 찾는 속도도 빨라집니다.
