## 왜 배우나

컴퓨터는 많은 판단을 참과 거짓으로 처리합니다. `if`, `while`, 필터 조건, 권한 검사,
테스트 코드의 기대값은 모두 어떤 문장이 참인지 거짓인지 묻는 구조입니다.

명제 논리는 이런 판단을 수학적으로 다루는 가장 작은 단위입니다. 코드를 더 정확하게
읽고, 조건이 언제 깨지는지 설명하려면 먼저 "판단 가능한 문장"을 작은 단위로 나누는
습관이 필요합니다.

예를 들어 다음 요구사항은 한 문장처럼 보이지만 실제로는 여러 명제가 섞여 있습니다.

- "사용자가 로그인했고, 이메일 인증을 마쳤으며, 정지된 계정이 아니면 글을 쓸 수 있다."
- "상품이 품절이 아니거나 예약 판매 중이면 구매 버튼을 보여 준다."
- "점수가 60점 이상이고 지각이 3회 미만이면 수료 처리한다."

이 문장들을 작은 명제로 나누면 코드 조건식, 테스트 케이스, 예외 상황을 더 차분하게
다룰 수 있습니다.

## 먼저 감 잡기

명제는 참 또는 거짓을 정할 수 있는 문장입니다.

- "7은 홀수이다"는 참인 명제입니다.
- "2 + 2 = 5"는 거짓인 명제입니다.
- "현재 사용자는 로그인했다"는 실행 시점의 상태를 보면 참/거짓을 정할 수 있으므로 명제입니다.
- "이 코드는 좋은가?"는 기준이 없으면 참/거짓을 정하기 어려워 명제로 보기 어렵습니다.
- "버튼을 예쁘게 만들어라"는 명령문이므로 명제가 아닙니다.

프로그래밍에서는 명제를 보통 불리언 값으로 다룹니다.

```ts
const isAdult = age >= 19;
const hasPermission = role === "admin" || role === "editor";
const canAccess = isAdult && hasPermission;
```

여기서 `age >= 19`, `role === "admin"`, `role === "editor"`는 각각 참 또는 거짓이 되는
작은 명제입니다. `canAccess`는 이 작은 명제들을 논리 연산으로 조합한 복합 명제입니다.

## 핵심 정의

명제는 참 또는 거짓 중 하나의 진리값을 갖는 문장입니다. 두 명제 P, Q가 있을 때 기본
논리 연산은 다음과 같습니다.

| 연산 | 의미 | 참이 되는 경우 |
| --- | --- | --- |
| $P \land Q$ | AND | P와 Q가 모두 참 |
| $P \lor Q$ | OR | P 또는 Q 중 하나 이상 참 |
| $\lnot P$ | NOT | P가 거짓 |

복합 명제는 작은 명제를 결합해서 만든 명제입니다.

- $P \land Q$: "P이고 Q이다"
- $P \lor Q$: "P이거나 Q이다"
- $\lnot P$: "P가 아니다"
- $(P \land Q) \lor R$: "P와 Q가 모두 참이거나, R이 참이다"

## 자주 헷갈리는 지점

일상어의 "또는"은 가끔 둘 중 하나만 고르는 느낌이지만, 수학과 프로그래밍의 OR은 보통
둘 다 참인 경우도 참으로 봅니다. "쿠폰이 있거나 포인트가 있으면 할인 가능"이라는
규칙에서 쿠폰과 포인트가 둘 다 있어도 할인 가능인 것과 같습니다.

명제와 의견도 자주 헷갈립니다.

- "이 함수는 10ms 안에 끝난다"는 측정 기준이 있으므로 명제입니다.
- "이 함수는 빠르다"는 빠름의 기준이 없으면 명제로 쓰기 어렵습니다.
- "사용자 이름은 비어 있지 않다"는 명제입니다.
- "사용자 이름을 잘 입력해라"는 명령이므로 명제가 아닙니다.

또 하나의 흔한 실수는 긴 조건식을 한 번에 읽으려는 것입니다. 다음 코드는 한 줄이지만
세 개의 명제가 들어 있습니다.

```ts
const canPublish = isLoggedIn && hasVerifiedEmail && !isSuspended;
```

초보자는 이 코드를 "글을 쓸 수 있는가?"라고만 읽기 쉽습니다. 하지만 디버깅할 때는
`isLoggedIn`, `hasVerifiedEmail`, `isSuspended`를 각각 따로 확인해야 어느 조건 때문에
실패했는지 알 수 있습니다.

### 예제를 해석하는 순서

복합 조건을 만나면 바로 진리표를 외우려고 하지 말고 다음 순서로 읽습니다.

1. 참/거짓을 정할 수 있는 가장 작은 문장을 찾습니다.
2. 각 문장에 P, Q, R 같은 이름을 붙입니다.
3. "그리고", "또는", "아니다"를 각각 $\land$, $\lor$, $\lnot$로 바꿉니다.
4. 코드라면 `&&`, `||`, `!`가 어느 작은 명제에 붙는지 괄호로 확인합니다.

예를 들어 "로그인했고 관리자이거나, 초대 링크가 있다"는 문장은 두 가지로 해석될 수
있습니다.

```ts
(isLoggedIn && isAdmin) || hasInvite
isLoggedIn && (isAdmin || hasInvite)
```

괄호 위치가 달라지면 초대 링크만 있는 사용자의 접근 가능 여부가 달라집니다. 명제 논리는
이런 애매함을 줄이기 위해 작은 판단과 결합 방식을 분리해서 보는 도구입니다.



## 코드로 읽기

논리 연산은 조건문을 구성하는 재료입니다.

```ts
if (isLoggedIn && hasAdminRole) {
  showAdminPanel();
}
```

여기서 `isLoggedIn`과 `hasAdminRole`은 각각 명제처럼 볼 수 있고, 전체 조건은 두 명제의
AND입니다.

조금 더 복잡한 요구사항도 같은 방식으로 분해할 수 있습니다.

```ts
const hasWritePermission = role === "admin" || role === "editor";
const canWritePost = isLoggedIn && hasWritePermission && !isSuspended;
```

- P: 사용자가 로그인했다.
- Q: 사용자는 관리자이거나 편집자이다.
- R: 사용자는 정지된 계정이다.
- 전체 조건: $P \land Q \land \lnot R$

이렇게 바꾸면 테스트 케이스도 자연스럽게 나옵니다. P, Q, R 중 하나씩 값을 바꿔 보면서
어느 조합에서 글쓰기가 허용되어야 하는지 확인할 수 있습니다.

## 전공에서 만나는 장면

명제 논리는 전공 과목의 큰 제목보다 더 작은 장면에서 반복됩니다.

- 자료구조: 스택이 비어 있는가, 큐가 가득 찼는가, 방문한 정점인가를 판단합니다.
- 데이터베이스: `WHERE active = true AND role = 'admin'`처럼 행을 남길 조건을 만듭니다.
- 테스트: "입력이 빈 문자열이면 에러가 난다"처럼 기대 결과를 참/거짓으로 검증합니다.
- 보안: 로그인 여부, 권한, 토큰 만료 여부를 조합해 접근 가능성을 판단합니다.
- 알고리즘: 반복문이 계속될 조건과 종료될 조건을 명제로 표현합니다.

복잡한 시스템도 결국 작은 판단들의 조합으로 동작합니다. 명제를 잘 나누면 코드의 의도를
설명하기 쉬워지고, 실패한 조건을 찾는 속도도 빨라집니다.
