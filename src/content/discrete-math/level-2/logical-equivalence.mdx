## 왜 배우나

조건식은 길어질수록 실수하기 쉽습니다. <strong>논리적 동치</strong>는 같은 참/거짓 결과를 내는 식을 더 읽기
쉬운 형태로 바꾸는 도구입니다. 코드를 바꾸더라도 조건의 의미가 같아야 하므로, 동치는
리팩터링의 안전장치처럼 쓰입니다.

## 먼저 감 잡기

`!(isAdmin && isActive)`는 "관리자이면서 활성 상태인 경우가 아니다"라는 뜻입니다. 같은 뜻을
조금 더 나누어 쓰면 `!isAdmin || !isActive`입니다. 둘은 모든 경우에서 같은 결과를 냅니다.

여기서 중요한 점은 식을 예쁘게 꾸미는 것이 아닙니다. 조건을 바꿔 썼을 때 <strong>허용되는 사용자와
차단되는 사용자가 그대로 유지되는지</strong>가 핵심입니다. 그래서 논리적 동치는 코드 리팩터링, 필터
반전, 테스트 조건 정리에서 자주 쓰입니다.

## 핵심 정의

두 논리식이 <strong>모든 P, Q 값에서 같은 진리값</strong>을 가지면 논리적으로 동치라고 합니다. 기호로는
$A \equiv B$처럼 씁니다.

자주 쓰는 동치:

- 이중 부정: $\lnot \lnot P \equiv P$
- 드모르간: $\lnot(P \land Q) \equiv \lnot P \lor \lnot Q$
- 드모르간: $\lnot(P \lor Q) \equiv \lnot P \land \lnot Q$
- 조건문: $P \to Q \equiv \lnot P \lor Q$
- 대우: $P \to Q \equiv \lnot Q \to \lnot P$

각 규칙을 자연어로 읽으면 다음과 같습니다.

- 이중 부정: "P가 아닌 것이 아니다"는 결국 P입니다.
- 드모르간 AND: "둘 다 참인 것은 아니다"는 "하나는 거짓이다"와 같습니다.
- 드모르간 OR: "둘 중 하나라도 참인 것은 아니다"는 "둘 다 거짓이다"와 같습니다.
- 조건문 변형: "P이면 Q"는 "P가 아니거나 Q이다"와 같습니다.
- 대우: "P이면 Q"는 "Q가 아니면 P도 아니다"와 같습니다.

동치 확인은 보통 <strong>진리표의 결과 열을 비교</strong>해서 합니다. 모든 줄이 같으면 동치이고, 한 줄이라도
다르면 동치가 아닙니다.

## 자주 헷갈리는 지점

동치는 *비슷해 보인다*가 아니라 <strong>모든 경우에서 결과가 같다</strong>는 뜻입니다. 예를 들어
$\lnot(P \land Q)$와 $\lnot P \land \lnot Q$는 같지 않습니다. P가 거짓이고 Q가 참일 때
앞의 식은 참이지만 뒤의 식은 거짓입니다.

조건문 $P \to Q$를 $\lnot P \lor Q$로 바꾸면, P가 거짓일 때 조건문이 참으로 처리되는 이유도
진리표로 확인할 수 있습니다.

또 하나의 흔한 실수는 <strong>대우와 역을 섞는 것</strong>입니다. $P \to Q$와 동치인 것은
$\lnot Q \to \lnot P$입니다. $Q \to P$는 역이고, 일반적으로 원래 조건문과 동치가 아닙니다.

## 코드로 읽기

상황: 관리자이면서 활성 상태인 사용자를 제외하고 싶습니다.

수학적 분해:

- P: 사용자가 관리자이다.
- Q: 사용자가 활성 상태이다.
- 제외 조건: $\lnot(P \land Q)$

```python
def should_hide(user):
    return not (user["is_admin"] and user["is_active"])

def should_hide_refactored(user):
    return (not user["is_admin"]) or (not user["is_active"])
```

두 함수는 같은 조건을 다른 모양으로 쓴 것입니다. 동치는 문법을 바꾸는 기술이 아니라 <strong>조건의 의미가 보존되는지 확인하는 도구</strong>입니다.

조건문 변형도 코드에서 자주 보입니다. 예를 들어 "회원이면 할인된다"라는 규칙을 검사할 때,
위반은 회원인데 할인이 없는 경우뿐입니다.

```python
def rule_is_valid(is_member, has_discount):
    return (not is_member) or has_discount
```

이 코드는 $P \to Q \equiv \lnot P \lor Q$를 그대로 옮긴 것입니다. P가 거짓인 경우에는 아직
규칙을 어긴 사례가 아니고, P가 참이면 Q가 참이어야 합니다.

필터 조건을 반전할 때도 드모르간 법칙이 유용합니다.

```python
def should_show(user):
    return user["is_admin"] and user["is_active"]

def should_hide(user):
    return (not user["is_admin"]) or (not user["is_active"])
```

`should_hide`는 `not should_show`를 풀어 쓴 형태입니다. 이렇게 풀어 쓰면 어느 조건 때문에
숨겨지는지 로그나 UI 메시지로 나누어 설명하기 쉬워집니다.
