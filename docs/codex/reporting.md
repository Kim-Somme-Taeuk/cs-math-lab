# Reporting Rules

Use Korean unless the user asks otherwise.

After completing any task, do not only report the result. Explain the work in a way that helps the project owner understand the project.

Final response must include:

## 1. What changed

- List the changed files.
- Explain each file's role in the project.
- Explain why that file needed to be changed.

## 2. How it works

- Explain the relevant flow in plain language.
- Mention important functions, components, hooks, routes, or data structures.
- Explain how the new code connects to the existing code.

## 3. Why this approach was chosen

- Explain the reasoning briefly.
- Mention tradeoffs if there were any.
- Do not over-engineer or describe unrelated alternatives.

## 4. How it was verified

- List the commands that were run.
- Say whether build, tests, lint, or manual checks passed.
- If something was not verified, say that clearly.

## 5. Remaining risks or next steps

- Mention only concrete risks or follow-up work.
- Do not invent unnecessary improvements.

## Rules

- Do not hide implementation details behind vague summaries.
- Do not say only "implemented" or "fixed".
- Keep explanations beginner-friendly but technically accurate.
- Keep the response concise unless the change is complex.
