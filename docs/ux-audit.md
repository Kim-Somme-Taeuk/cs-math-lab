# CS Math Lab UX Audit

검토일: 2026-06-10

검토 범위:

- 페이지: 홈, 로드맵, 과목 상세, 챕터 상세
- 과목: 이산수학, 선형대수, 미적분, 확률통계 예정 상태
- 대표 챕터: 이산수학 `logic`, `sets`, `recursion-recurrences`; 선형대수 `vectors`, `determinants`, `gradient-jacobian-intro`; 미적분 `calculus-functions-graphs`, `gradient-descent`, `numerical-integration`
- 컴포넌트: 챕터 네비게이션, `MultipleChoiceQuiz`, `GeneratedReviewQuiz`, 이해도 체크, 인터랙티브 playground, 개인화 추천 패널, AI 챗봇
- 확인 방식: 코드 구조 검토와 Playwright 기반 렌더링 확인

## 요약

- 전체 UX 상태
  - 기본 구조는 명확하다. 홈 → 로드맵 → 과목 → 챕터로 이어지는 큰 흐름이 있고, active/planned 상태와 ready 챕터도 대체로 구분된다.
  - 강점은 컴공 맥락의 설명, 챕터별 실험 컴포넌트, 퀴즈와 이해도 체크의 반복 배치다.
  - 가장 큰 약점은 “처음 방문자가 왜 이 순서로 가야 하는지”와 “개인화/AI가 어떤 근거로 추천하는지”가 충분히 설명되지 않는 점이다.
  - 챕터 품질과 길이 편차가 크다. 일부 이산수학 챕터는 300줄 이상으로 촘촘하지만, 일부 미적분 챕터는 80줄 내외로 압축되어 같은 사이트 안에서 학습 밀도가 다르게 느껴진다.

- 가장 큰 문제 3개
  - 개인화 추천과 AI 코치가 신뢰 근거를 충분히 보여 주지 않는다.
  - 과목/Level/챕터 관계는 보이지만 선행 관계와 다음 학습 이유가 약하다.
  - 모바일 챕터 화면에서 AI 챗봇과 챕터 이동 버튼이 모두 고정되어 본문과 퀴즈 흐름을 방해할 수 있다.

- 바로 고칠 것 5개
  - 홈의 “첫 챕터 시작”이 왜 `명제와 논리`인지 설명한다.
  - 과목 카드와 챕터 카드에 prerequisites 또는 “다음에 배우는 이유”를 최소 한 줄로 노출한다.
  - 개인화 추천 카드에 추천 근거 태그를 표시한다.
  - 모바일 챕터에서 AI 챗봇과 챕터 이동 버튼의 위치 충돌 가능성을 줄인다.
  - 퀴즈 채점 후 오답 원인별 복습 링크나 관련 섹션 링크를 제공한다.

## 페이지별 리뷰

### 홈

홈은 `src/app/page.tsx`와 `ReadySubjectSummary`로 구성된다. 첫 화면에는 서비스 목적, “첫 챕터 시작”, “로드맵 보기”, 공개 학습 트랙 요약이 있다. 처음 방문자가 바로 시작할 수 있는 CTA는 있으나, 왜 이산수학의 `logic`부터 시작하는지 설명이 없어 사용자가 선형대수나 미적분 관심자로 들어왔을 때 경로 선택이 불명확하다.

공개 챕터 수와 과목 상태는 빠르게 보인다. 다만 “진행 중”과 “예정”은 과목 상태이고, 챕터의 “공개 중/준비 중/예정”과 다른 의미인데 홈에서는 이 차이를 자세히 설명하지 않는다.

### 로드맵

로드맵은 `src/app/roadmap/page.tsx`와 `PersonalizedPathPanel`이 핵심이다. 과목별 공개 수, Level 수, Level 상태를 한눈에 보여 주는 점은 좋다. 그러나 개인화 설정 패널이 로드맵 카드보다 먼저 크고 강하게 노출되어, 처음 온 사용자는 “설정부터 해야 하나?”라고 느낄 수 있다.

확률통계는 planned 과목으로 분명히 표시되지만, 카드 전체가 과목 페이지로 이동한다. 예정 과목을 눌렀을 때 공개 챕터가 없다는 안내는 나오지만, 로드맵 카드 단계에서 “학습 불가”와 “계획 보기 가능”의 차이가 더 명확하면 좋다.

### 과목 페이지

과목 페이지는 `src/app/subjects/[subjectSlug]/page.tsx`에서 Level 섹션과 챕터 카드를 렌더링한다. 추천 시작점은 유용하지만 항상 첫 ready 챕터만 고른다. 이미 다른 과목을 공부했거나, 사용자가 로드맵에서 특정 Level을 보던 상황을 반영하지 않는다.

챕터 카드에는 order, title, status, `csConnection`, 미니 비주얼이 있다. 카드 밀도는 적절하지만 prerequisites, 소요 시간, 난이도, 완료 여부가 보이지 않아 어떤 챕터를 먼저 열어야 할지 스스로 판단하기 어렵다.

### 챕터 페이지

챕터 페이지는 `src/app/chapters/[slug]/page.tsx`가 본문, 섹션 네비게이션, 같은 Level 챕터 이동, 이전/다음 링크, AI 챗봇을 배치한다. 데스크톱에서는 우측 네비게이션이 학습 흐름을 잘 보조한다. 모바일에서는 섹션 이동과 챕터 이동을 floating details로 숨겨 공간은 아끼지만, 동시에 AI 챗봇 버튼도 떠 있어 고정 UI가 많다.

본문은 MDX의 공통 섹션 구조를 따른다. 다만 챕터마다 길이와 깊이 차이가 크고, 마지막 CTA가 “다음 챕터로 가기”인데 실제 href가 과목 페이지인 경우도 있어 기대와 결과가 어긋난다.

## 학습 흐름 리뷰

- 처음 방문자 흐름
  - 홈에서 “첫 챕터 시작”을 누르면 `logic`으로 간다. 빠르지만 개인 목표별 대안이 약하다.

- 과목 선택 흐름
  - 홈과 로드맵 모두 과목 선택은 쉽다. 하지만 과목 카드에서 active/planned의 행동 차이가 충분히 예측되지 않는다.

- Level 진행 흐름
  - Level 단위로 챕터가 묶여 있어 구조는 좋다. 그러나 Level 간 선행 관계, 다음 Level 진입 조건, 추천 순서의 이유가 부족하다.

- 챕터 완료 후 흐름
  - `NextChapterButton`과 우측 네비게이션이 있다. 다만 완료 체크는 로드맵 개인화 패널 안에 있고 챕터 끝에는 없어, 실제 학습 완료 행동이 분리되어 있다.

## 컴포넌트 리뷰

- 퀴즈
  - `MultipleChoiceQuiz`는 진행 상황, 전체 채점, 해설 보기, 이해/어려움 피드백을 제공한다.
  - 장점: 정답과 해설 저장, 여러 문제 페이지네이션, 오답 후 “비슷한 문제 다시 풀기” 일부 지원.
  - 약점: 해설은 기본적으로 접혀 있고, 오답 선택지별 맞춤 해설은 일반 문장에 가깝다. 오답 후 관련 본문 섹션으로 돌아가는 링크가 없다.

- 인터랙티브
  - 대부분 개념 이해를 돕는 구조다. `TruthTablePlayground`, `LinearAlgebraPlayground`, `GradientDescentPlayground`처럼 입력을 바꾸면 결과가 바뀐다.
  - 약점: 일부 playground는 여러 모드에서 같은 결과 카드 전체를 보여 주기 때문에 현재 챕터와 직접 관련 없는 카드까지 노출된다. SVG 중심 컴포넌트는 시각적 변화의 텍스트 대체가 약하다.

- 개인화 패널
  - 사용자의 목표/수준/선호와 로컬 학습 기록을 읽어 추천 경로를 만든다.
  - 약점: 추천 근거와 선행 관계를 카드에서 설명하지 않아 “개인화”가 실제로 작동하는지 체감하기 어렵다.

- AI 코치
  - 로드맵의 AI 코치 메모와 챕터별 AI 챗봇이 있다.
  - 약점: AI 기능은 강하게 보이지만, 기본 학습 흐름과 얼마나 연결되는지 설명이 부족하다. 모바일에서는 floating 버튼이 학습 UI를 가릴 수 있다.

- 카드/버튼/배지
  - 전체적으로 둥근 사각형 카드, slate/teal/amber 배지 스타일이 일관적이다.
  - 약점: `진행 중`, `공개 중`, `준비 중`, `예정`의 의미가 비슷하게 보이고, “자세히 보기”가 링크 카드 내부에 별도 버튼처럼 보여 클릭 범위가 혼동될 수 있다.

## 모바일 리뷰

모바일에서 가로 overflow는 대표 화면에서 확인되지 않았다. 카드와 퀴즈는 단일 컬럼으로 잘 줄어든다. 하지만 챕터 페이지는 AI 챗봇 버튼과 챕터 이동 버튼이 모두 fixed로 떠서 하단 영역의 주의를 빼앗는다. 긴 챕터에서 퀴즈 버튼, 해설 버튼, floating 버튼이 함께 나타나면 클릭 목표가 많아진다.

개인화 추천 카드의 모바일 카드 전체 링크와 완료 체크 버튼이 같은 카드 안에 겹친다. z-index로 버튼 클릭은 가능하지만 사용자가 “카드를 눌러 학습”과 “체크로 완료”를 구분하기 어렵다.

## 접근성 리뷰

기본 HTML 구조는 `main`, `nav`, `fieldset`, `legend`, `aria-label`을 일부 사용하고 있어 출발점은 좋다. 다만 인터랙티브 컴포넌트의 상태 변화가 `aria-live`로 알려지지 않고, SVG 그래프의 변화가 텍스트 결과와 충분히 연결되지 않는 경우가 있다.

키보드 포커스 스타일은 Tailwind 기본 outline에 의존한다. 많은 버튼이 커스텀 배경/테두리를 쓰므로 명시적인 `focus-visible` 스타일이 필요하다. floating details summary는 모바일 챕터 이동의 핵심이지만 열림/닫힘 상태와 내부 메뉴 도달성이 더 분명해야 한다.

## 콘텐츠 가독성 리뷰

본문은 컴공 사례와 코드 예시를 섞어 초보자 친화적이다. `math-content`에서 line-height, code, pre, table, KaTeX overflow 처리를 해 두어 기본 가독성은 좋다.

다만 챕터 길이 편차가 크다. 예를 들어 `logic.mdx`는 353줄, `recursion-recurrences.mdx`는 390줄인데, `calculus-functions-graphs.mdx`는 86줄, `gradient-descent.mdx`는 77줄이다. 같은 Level/챕터 카드로 들어왔을 때 학습량 예측이 어렵다.

## Findings

- ID: UX-001
- Severity: High
- 위치: 홈, `src/app/page.tsx`
- 문제 설명: “첫 챕터 시작”이 곧바로 `/chapters/logic`으로 이동하지만, 왜 이 챕터가 첫 시작점인지 설명하지 않는다.
- 사용자 영향: 선형대수나 미적분 목적의 사용자는 자신의 관심과 맞는 시작점을 찾기 전에 이탈할 수 있다.
- 재현 또는 확인 방법: 홈 첫 화면에서 CTA 문구와 링크 목적지를 확인한다.
- 최소 수정 제안: CTA 주변에 “수학이 막막하면 이산수학 Level 1부터” 같은 이유 문구와 “과목별로 시작” 보조 링크를 추가한다.

- ID: UX-002
- Severity: Medium
- 위치: 홈, `ReadySubjectSummary`
- 문제 설명: 공개 챕터 수는 보이지만 각 과목에서 무엇을 먼저 배우는지 요약되지 않는다.
- 사용자 영향: 공개량은 알지만 학습 선택 기준은 얻기 어렵다.
- 재현 또는 확인 방법: 홈의 “공개 학습 트랙” 카드에서 Level별 count만 확인한다.
- 최소 수정 제안: 각 과목 카드에 첫 ready 챕터 또는 Level 1 핵심 키워드 2개를 노출한다.

- ID: UX-003
- Severity: Medium
- 위치: 홈, 대주제 카드
- 문제 설명: active 과목과 planned 과목이 모두 같은 링크 카드로 보인다.
- 사용자 영향: 예정 과목도 바로 학습 가능한 것으로 오해할 수 있다.
- 재현 또는 확인 방법: 홈의 확률통계 카드가 active 과목과 같은 클릭 패턴인지 확인한다.
- 최소 수정 제안: planned 카드는 “계획 보기”로 라벨을 바꾸고, 학습 CTA와 시각 톤을 더 약하게 만든다.

- ID: UX-004
- Severity: Medium
- 위치: 로드맵, `PersonalizedPathPanel`
- 문제 설명: 개인화 설정 패널이 로드맵보다 먼저 크게 나타나며, 설정하지 않으면 안 되는 것처럼 보인다.
- 사용자 영향: 처음 방문자가 로드맵 탐색 전에 설정 부담을 느낄 수 있다.
- 재현 또는 확인 방법: `/roadmap` 첫 방문 상태에서 목표/수준/선호 버튼들이 상단을 차지하는지 확인한다.
- 최소 수정 제안: 개인화 패널을 접힘 상태 또는 “선택 사항” 문구와 함께 로드맵 아래로 낮춘다.

- ID: UX-005
- Severity: High
- 위치: 로드맵, 개인화 추천
- 문제 설명: 추천 경로가 어떤 기준으로 정렬되었는지 카드에 표시되지 않는다.
- 사용자 영향: 추천을 신뢰하기 어렵고 임의 목록처럼 느껴진다.
- 재현 또는 확인 방법: 프로필 저장 후 추천 카드에 이유/태그/선행 관계가 있는지 확인한다.
- 최소 수정 제안: 각 추천 카드에 “기초 보강”, “선행: logic”, “최근 오답: 집합” 같은 근거 태그를 추가한다.

- ID: UX-006
- Severity: Medium
- 위치: 로드맵, AI 코치 메모
- 문제 설명: AI 코치 메모 생성 버튼은 있으나 어떤 데이터를 사용하고 어떤 한계가 있는지 화면 안에서 충분히 알 수 없다.
- 사용자 영향: 사용자는 AI 메모가 정확한지, 개인정보나 학습 기록을 어떻게 쓰는지 판단하기 어렵다.
- 재현 또는 확인 방법: `/roadmap`에서 AI 코치 메모 영역을 확인한다.
- 최소 수정 제안: “프로필, 완료, 퀴즈 기록 기준” 같은 짧은 근거 설명과 fallback/cache 표시를 항상 보이게 한다.

- ID: UX-007
- Severity: Medium
- 위치: 로드맵 과목 카드
